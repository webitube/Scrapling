## Code Quality Pipeline

Triggered on push/PR to `main` or `dev` (excluding markdown-only changes).

### Tools & Steps

| Step | Tool | Purpose |
|------|------|---------|
| 1 | **Bandit** | Security linting (`.bandit.yml` config) |
| 2 | **Ruff (linter)** | Style and correctness checks (`ruff.toml` config) |
| 3 | **Ruff (formatter)** | Formatting validation |
| 4 | **Vermin** | Python 3.10+ compatibility check |
| 5 | **Mypy** | Static type checking |
| 6 | **PyRight** | Static type checking (alternative) |

### Run Locally

```bash
# Bandit
bandit -r -c .bandit.yml scrapling/

# Ruff lint
ruff check scrapling/

# Ruff format check
ruff format --check scrapling/

# Vermin
vermin -t=3.10- --violations --eval-annotations --no-tips scrapling/

# Mypy
mypy scrapling/

# PyRight
pyright scrapling/
```

### Configuration Files

- `ruff.toml` вЂ” Ruff linter and formatter settings
- `.bandit.yml` вЂ” Bandit security skips
- `.pre-commit-config.yaml` вЂ” Pre-commit hook definitions

## Testing Pipeline

Triggered on push/PR to `main` or `dev`. Runs on macOS across Python 3.10вЂ“3.13 via Tox.

### Tox Configuration

Tox runs three sequential test phases to avoid conflicts:

1. **Browser tests** вЂ” `DynamicFetcher` and `StealthyFetcher` tests (no parallelization)
2. **Async tests** вЂ” Tests marked with `asyncio` (no parallelization)
3. **Remaining tests** вЂ” All other tests (parallelized with `pytest-xdist`)

### Run Locally

```bash
# Install tox
pip install tox

# Run all environments (all Python versions + pre-commit)
tox

# Run a specific Python version
tox -e py312

# Run only pre-commit checks
tox -e pre-commit
```

### Run pytest Directly

```bash
# All tests
pytest --cov=scrapling --cov-report=xml -v

# With parallelization
pytest -n auto -v

# Specific test file
pytest tests/parser/test_selector.py -v
```

### Configuration Files

- `tox.ini` вЂ” Tox environments and test commands
- `pytest.ini` вЂ” pytest settings (strict asyncio mode, verbose output)
- `tests/requirements.txt` вЂ” Test dependencies

## Docker Build & Push

Triggered when a PR is merged to `main` or manually via `workflow_dispatch`.

### Image Registries

| Registry | Image Name |
|----------|-----------|
| Docker Hub | `pyd4vinci/scrapling` |
| GitHub Container Registry | `ghcr.io/d4vinci/scrapling` |

### Build Details

- **Base image:** `python:3.12-slim-trixie`
- **Package manager:** `uv` (faster than pip)
- **Platforms:** `linux/amd64`, `linux/arm64`
- **Exposed port:** `8000` (MCP server HTTP transport)
- **Entrypoint:** `uv run scrapling`

### Tag Strategy

Automatic tags generated by `docker/metadata-action`:
- Branch name
- PR number
- Semantic version (`v1.2.3`, `1.2`, `1`)
- `latest` (on default branch)

### Required Secrets

| Secret | Purpose |
|--------|---------|
| `DOCKER_USERNAME` | Docker Hub login |
| `DOCKER_PASSWORD` | Docker Hub login |
| `CONTAINER_TOKEN` | GitHub Container Registry login |

### Build Locally

```bash
# Build the image
docker build -t scrapling:local .

# Run the container
docker run --rm scrapling:local --help

# Run with MCP server on port 8000
docker run --rm -p 8000:8000 scrapling:local mcp-server
```

### Dockerfile Optimization

The Dockerfile uses layer caching to speed up rebuilds:
1. `pyproject.toml` is copied first so dependency installation is cached
2. `uv` cache is mounted for faster installs
3. APT cache is mounted for system dependencies
4. Static version in `pyproject.toml` (not dynamic) enables better layer caching

Triggered automatically when a PR is merged to `main`.

### Release Workflow

```
PR merged to main
    в”‚
    в”њв”Ђв†’ Extract version from PR title (must start with 'v', e.g., 'v0.4.9')
    в”‚
    в”њв”Ђв†’ Create GitHub Release
    в”‚     в””в”Ђв†’ PR body becomes release notes
    в”‚
    в””в”Ђв†’ Build and publish to PyPI
          в”њв”Ђв†’ python -m build --sdist --wheel
          в””в”Ђв†’ pypa/gh-action-pypi-publish (trusted publishing)
```

### Steps to Release

1. Develop changes on a feature branch
2. Open a PR with the title in version format: `v0.X.Y`
3. Write release notes in the PR body
4. Merge the PR вЂ” the workflow handles the rest

### Requirements

- PR title **must** start with `v` (e.g., `v0.4.9`, `v1.0.0`)
- PyPI environment uses trusted publishing (`id-token: write`)
- No manual PyPI credentials needed

## MCP Server

The MCP (Model Context Protocol) server exposes Scrapling's web scraping tools to AI clients like Claude Desktop, Cursor, or any MCP-compatible interface.

### Install

```bash
# Install MCP dependencies
pip install "scrapling[ai]"

# Install browser dependencies
scrapling install
```

### Run via CLI

```bash
# Stdio transport (default, for AI clients)
scrapling mcp

# HTTP transport (for web-based clients)
scrapling mcp --http
```

### Run Programmatically

```python
from scrapling.core.ai import ScraplingMCPServer

server = ScraplingMCPServer()

# Stdio transport
server.serve(http=False)

# HTTP transport
server.serve(http=True, host="0.0.0.0", port=8000)
```

### Run via Docker

```bash
# Stdio transport
docker run --rm pyd4vinci/scrapling mcp

# HTTP transport (port 8000)
docker run --rm -p 8000:8000 pyd4vinci/scrapling mcp --http
```

### Configure in an AI Client

**Claude Desktop** вЂ” add to `claude_desktop_config.json`:
```json
{
  "mcpServers": {
    "scrapling": {
      "command": "python",
      "args": ["-m", "scrapling", "mcp"]
    }
  }
}
```

**HTTP-based clients** вЂ” point to `http://localhost:8000`.

### Configure with mcp-proxy

mcp-proxy allows you to run multiple MCP servers from a single configuration file.

**Stdio transport** (recommended for AI clients):
```json
{
  "servers": {
    "scrapling": {
      "command": "scrapling",
      "args": ["mcp"],
      "env": {}
    }
  }
}
```

**HTTP transport**:
```json
{
  "servers": {
    "scrapling": {
      "command": "scrapling",
      "args": ["mcp", "--http", "--host", "0.0.0.0", "--port", "8000"],
      "env": {}
    }
  }
}
```

**MCP Proxy**
```json
{
    "mcpServers": {
		"scrapling": {
		  "command": "scrapling",
		  "args": ["mcp"],
		  "env": {}
		},
    }
}
```

Launch with:
```bash
mcp-proxy --config config.json
```

### Available Tools

| Tool | Description |
|------|-------------|
| `get` | Fast HTTP requests with browser impersonation |
| `bulk_get` | Async parallel HTTP requests |
| `fetch` | Dynamic content via Chromium/Chrome |
| `bulk_fetch` | Async parallel browser fetches |
| `stealthy_fetch` | Bypass Cloudflare and anti-bot systems |
| `bulk_stealthy_fetch` | Async parallel stealth fetches |
| `screenshot` | Capture PNG/JPEG screenshots |
| `open_session` | Create persistent browser sessions |
| `close_session` | Close browser sessions |
| `list_sessions` | List active sessions |

### Configuration Files

- `scrapling/core/ai.py` вЂ” MCP server implementation
- `scrapling/cli.py` вЂ” CLI entry point (`scrapling mcp`)
- `docs/ai/mcp-server.md` вЂ” Full MCP documentation
- `docs/api-reference/mcp-server.md` вЂ” API reference

## Pre-commit Hooks

Three hooks run on every commit:

| Hook | Tool | Purpose |
|------|------|---------|
| 1 | **Bandit** | Security vulnerability scanning |
| 2 | **Ruff** | Linting with auto-fix |
| 3 | **Ruff Format** | Code formatting |
| 4 | **Vermin** | Python version compatibility |

### Setup & Run

```bash
# Install pre-commit
pip install pre-commit

# Install git hooks
pre-commit install

# Run manually on all files
pre-commit run --all-files

# Run on staged files only
pre-commit run
```

## Troubleshooting

### Common Issues

**Playwright browsers not found**
```bash
playwright install chromium
playwright install-deps chromium
```

**Tests fail with import errors**
```bash
pip install -e ".[all]"
pip install -r tests/requirements.txt
```

**Docker build fails on ARM**
Ensure Buildx is set up with QEMU for multi-arch:
```bash
docker buildx create --use --name multiarch
docker buildx inspect --bootstrap
```

**PyRight / Mypy report false positives**
Ensure `lxml-stubs` is installed:
```bash
pip install lxml-stubs
```

**Pre-commit hooks fail**
```bash
pre-commit clean
pre-commit install
```

### CI/CD Concurrency

Both the tests and code quality workflows use concurrency groups to cancel in-progress runs when new commits are pushed to the same branch:

```yaml
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true
```

This prevents redundant CI runs and saves resources.
